# Technical Documentation

## Project Overview

This project is an end-to-end e-commerce product analytics system built with Python, DuckDB, SQL, Streamlit, and AI-assisted insight generation. It simulates realistic retail customer behaviour, loads the data into a local analytical warehouse, transforms it through a Bronze/Silver/Gold medallion architecture, and exposes business-ready metrics through a Streamlit dashboard.

The system is designed to support product analytics questions such as:

- Where do users drop out of the shopping funnel?
- Did the checkout experiment improve conversion?
- Which segments should the product team prioritise?
- What is the estimated commercial impact?
- What should be communicated to senior stakeholders?

## Technology Stack

- Python: orchestration, data generation, statistics, AI agent logic
- DuckDB: local SQL OLAP warehouse
- SQL: Bronze, Silver, and Gold transformations
- Pandas: synthetic data generation and dataframe handling
- Streamlit: dashboard interface
- Plotly: interactive charts
- SciPy: two-proportion z-test and confidence intervals
- OpenAI API: insight generation when available
- Anthropic Claude API: validation when available
- Markdown docs: lightweight RAG knowledge base
- Rule-based fallback: keeps the app usable without API keys

## Data Generation

Synthetic data is generated by `src/generate_data.py`. The generator creates 90 days of realistic e-commerce grocery/app behavioural data.

Generated entities:

- Users: 50,000 users
- Products: 600 products
- Sessions: customer visits across app/web
- Events: funnel events from homepage to payment completion
- Orders: completed purchases with revenue and basket values
- Experiment assignments: user-level control/variant assignment

Key dimensions:

- Device type: mobile, desktop, tablet
- Region: London, South East, Midlands, North West, Scotland, Wales
- Customer type: new, returning
- Loyalty flag: true, false
- Channel: app, web

Funnel event sequence:

```text
homepage_view
search
product_view
add_to_basket
checkout_start
payment_complete
```

The synthetic experiment is intentionally designed so the variant improves conversion overall, while performance varies by segment. This creates realistic analytical nuance for the dashboard and AI assistant.

## Medallion Architecture

### Bronze Layer

Bronze tables preserve raw source data with minimal transformation.

SQL file:

```text
sql/bronze/create_bronze_tables.sql
```

Tables:

- `raw_users`
- `raw_sessions`
- `raw_events`
- `raw_orders`
- `raw_products`
- `raw_experiment_assignments`

Purpose:

- Load raw CSV files into DuckDB
- Keep a traceable copy of source data
- Avoid business logic in the ingestion step

### Silver Layer

Silver tables clean and standardise the raw data.

SQL file:

```text
sql/silver/create_silver_tables.sql
```

Tables:

- `clean_users`
- `clean_sessions`
- `clean_events`
- `clean_orders`
- `clean_products`
- `clean_experiment_assignments`

Transformations:

- Cast fields to correct data types
- Standardise categorical values
- Remove duplicate identifiers with window functions
- Validate non-null primary identifiers
- Prepare consistent entities for analytics

### Gold Layer

Gold tables serve dashboard and export use cases.

SQL files:

```text
sql/gold/create_gold_kpi_tables.sql
sql/gold/create_gold_funnel_tables.sql
sql/gold/create_gold_experiment_tables.sql
sql/gold/create_gold_segment_tables.sql
```

Tables:

- `gold_daily_kpis`
- `gold_funnel_metrics`
- `gold_experiment_results`
- `gold_segment_performance`
- `gold_product_performance`

Purpose:

- Provide business-ready KPI marts
- Support dashboard pages
- Export data to Tableau Public
- Keep metric definitions governed in SQL

## Pipeline Execution

Run the full pipeline from the project root:

```bash
python src/generate_data.py
python src/etl_bronze.py
python src/etl_silver.py
python src/etl_gold.py
streamlit run app/streamlit_app.py
```

Pipeline responsibilities:

- `generate_data.py`: writes synthetic CSVs to `data/raw/`
- `etl_bronze.py`: loads raw CSVs into DuckDB Bronze tables
- `etl_silver.py`: creates cleaned Silver tables
- `etl_gold.py`: creates Gold marts and exports Tableau-ready CSVs

## Metric Definitions

### Revenue

Total completed order revenue.

```text
SUM(clean_orders.revenue)
```

### Orders

Count of completed orders.

```text
COUNT(clean_orders.order_id)
```

### Conversion Rate

Completed orders divided by sessions.

```text
orders / sessions
```

### Average Order Value

Revenue divided by completed orders.

```text
revenue / orders
```

### Active Users

Distinct users with at least one session in the selected period or filter context.

```text
COUNT(DISTINCT clean_sessions.user_id)
```

### Funnel Step Conversion

Sessions reaching a funnel step divided by sessions reaching the previous step.

### Drop-off Rate

The inverse of step conversion.

```text
1 - step_conversion_rate
```

### Commercial Impact Estimate

Estimated value of experiment uplift applied to selected traffic.

```text
experiment_uplift * sessions * average_order_value
```

This estimate is directional and should be interpreted as a business sizing metric, not audited finance.

## Experimentation Methodology

The A/B test compares:

- Control: existing checkout experience
- Variant: reduced checkout friction

Primary success metric:

```text
payment_complete sessions / total sessions
```

Statistical method:

- Two-proportion z-test
- 95% confidence interval
- p-value threshold of 0.05

Implemented in:

```text
src/experiment_analysis.py
```

Important interpretation rule:

The global experiment result may differ from filtered segment results. This is intentional and analytically realistic. Segment-level views are useful for diagnosis and prioritisation, but should be interpreted carefully if sample sizes are smaller.

## AI Agent Design

AI logic is implemented in:

```text
src/ai_agents.py
```

Agents:

- DataQualityAgent: checks row counts, duplicate events, missing timestamps, and suspicious funnel drops
- ExperimentAnalystAgent: explains statistical results and confidence
- InsightAgent: explains dashboard metrics in plain English
- RecommendationAgent: prioritises product recommendations
- ValidationAgent: uses Claude when available to challenge or validate recommendations
- ReportAgent: combines insight, validation, data quality, and recommendation output

## LLM Governance

The system follows a strict metric governance rule:

```text
Python and SQL calculate metrics.
LLMs explain metrics.
LLMs do not invent numbers.
```

OpenAI is used for first-pass insight generation when `OPENAI_API_KEY` is available.

Claude is used for validation when `ANTHROPIC_API_KEY` is available.

If API keys are unavailable, the system falls back to deterministic rule-based summaries.

## RAG Knowledge Base

The RAG layer retrieves context from markdown files in `docs/`.

Documents:

- `product_analytics_glossary.md`
- `experimentation_playbook.md`
- `stakeholder_reporting_guide.md`
- `retail_product_context.md`

Retrieval logic is implemented in:

```text
src/rag_index.py
```

The current implementation uses lightweight keyword retrieval so the app remains simple and runnable locally. The project includes `chromadb` in the dependency list so the retrieval layer can be upgraded to vector search later.

## Data Quality Checks

The DataQualityAgent reports:

- Clean user row count
- Clean session row count
- Clean event row count
- Clean order row count
- Duplicate event count
- Missing event timestamp count
- Suspicious funnel drop checks

These checks are displayed in the AI Insight Assistant.

## Error Handling and Fallbacks

The app is designed to remain usable when API keys are missing.

Fallback behaviours:

- Missing OpenAI key: rule-based insight summary
- Missing Claude key: rule-based validation
- Missing pipeline outputs: dashboard shows an error instructing the user to run pipeline commands

## Known Limitations

- Synthetic data approximates realistic behaviour but is not real customer data.
- The RAG implementation is lightweight keyword retrieval rather than full embedding search.
- Segment experiment results should be treated as diagnostic unless the segment hypothesis is pre-planned.
- Commercial impact is an estimate based on conversion uplift and average order value.
- The app is designed for local portfolio demonstration rather than production deployment.
